# Wild Offscreen

**Wild Offscreen** — расширение для SillyTavern, которое генерирует автономные закадровые события для второстепенных персонажей (NPC): что происходит в их жизни пока разворачивается основная сцена. События генерируются отдельным вызовом AI, хранятся по каждому персонажу и чату, и инжектируются в контекст — так основная модель всегда знает, чем занимались все NPC за кадром.

## Как это работает — общий принцип

1. Вы сканируете лорбук или добавляете NPC вручную. Расширение считывает записи и сохраняет имена, описания и поисковые ключи персонажей.
2. Каждые N сообщений бота (настраивается) расширение вызывает отдельную AI-модель через выбранный Connection Profile.
3. В промпт отправляются: персонаж бота, персона пользователя, информация о текущей сцене (дата, время, локация, присутствующие персонажи) и блок по каждому NPC с описанием, заметками, закреплёнными фактами и историей событий.
4. AI возвращает по одной строке на каждого NPC в формате `NPC1: локация | масштаб | описание события`. NPC, которые находятся в активной сцене, получают статус `in-scene` — событие для них не генерируется.
5. Результаты парсятся, сохраняются и инжектируются в контекст основного чата в виде блока `[OFF-SCREEN NPC UPDATES]`.

## Настройка лорбука

Расширение сканирует записи лорбука в поисках карточек NPC. Режим сканирования выбирается в настройках (**Scan entries at position**):

### Режим `before_char` (по умолчанию)

Запись считается карточкой NPC если выполняется **хотя бы одно** из условий:

- Поле `position` равно `0` или `"before_char"`
- В массиве `key` есть строка `character`

**Рекомендуемая структура записи в лорбуке:**

```json
{
  "key": ["Савелий", "Парфёнов", "character"],
  "comment": "Савелий Парфёнов",
  "content": "Полное описание персонажа...",
  "position": 0,
  "constant": true
}
```

- Поле `comment` становится отображаемым именем NPC
- Поле `content` — описанием, которое отправляется в промпт
- Массив `key` сохраняется как поисковые ключи и используется для определения присутствия в сцене

### Режим `after_char`

Сканирует записи с `position: 1` или `position: "after_char"`. Полезно если вы храните описания отношений или биографии персонажей в отдельных записях и хотите использовать их как источник для NPC.

### Поисковые ключи и регулярные выражения

Все ключи из массива `key` сохраняются вместе с NPC и используются в двух местах:

- **Определение присутствия в сцене** — совпадение имени NPC со списком персонажей в инфоблоке
- **Поиск упоминаний в истории чата** — нахождение сообщений, где этот NPC участвовал в сцене

Ключи могут быть обычными строками или регулярными выражениями в формате `/паттерн/флаги`:

```
"Парфёнов"                                        ← обычное вхождение подстроки
/\bParfyonov\b/i                                  ← регекс
/(?:^|[^а-яёА-ЯЁ])Савелий(?=[^а-яёА-ЯЁ]|$)/i   ← регекс с границей слова
```

Это позволяет NPC с английским именем в лорбуке корректно определяться в сцене, где его имя написано по-русски — достаточно добавить соответствующие ключи в массив `key`.

## Формат инфоблока

Расширение считывает информацию о сцене из стилизованного HTML-блока, который основная модель пишет в начале каждого сообщения. Блок должен строго соответствовать формату.

### Обязательный формат

```html
<div style="padding: 10px; border-left: 3px solid var(--SmartThemeQuoteColor); background-color: ...; ...">
2008/11/12 • 14:15 • Тюменское ГУВД, кабинет Парфёнова • Савелий Парфёнов, Андрей Кравцов, Лилит
</div>
```

Содержимое блока делится по символу `•` (U+2022):

| Сегмент | Содержимое | Пример |
|---------|-----------|--------|
| `parts[0]` | Дата в формате `ГГГГ/ММ/ДД` | `2008/11/12` |
| `parts[1]` | Время в формате `ЧЧ:ММ` | `14:15` |
| `parts[2]` | Текущая локация | `Тюменское ГУВД, кабинет Парфёнова` |
| `parts[3]` | Персонажи в сцене через запятую | `Савелий Парфёнов, Андрей Кравцов, Лилит` |

### Нормализация даты

Если основная модель пишет месяц словом (`2008/Ноябрь/12`), расширение автоматически приводит его к числовому формату `2008/11/12`. Поддерживаются русский (все падежи) и английский языки.

### Что извлекается из инфоблока

- **Дата и время** → отображаются в интерфейсе и отправляются в промпт вместе с меткой времени суток (утро, день, вечер, ночь)
- **Локация** → передаётся в промпт чтобы модель знала где происходит активная сцена
- **Список персонажей** → сопоставляется со всеми зарегистрированными NPC для определения кто `[IN SCENE]`, а кто `[OFFSCREEN]`

### Логика определения присутствия в сцене

Для каждого NPC расширение проверяет совпадение с `parts[3]` в три уровня:

1. **Прямое совпадение** — `"Парфёнов"` найдёт `"Савелий Парфёнов"`
2. **Обычный ключ** — любой ключ из массива `key` лорбука как подстрока в списке персонажей
3. **Ключ-регекс** — любой `/паттерн/флаги` ключ тестируется против строки со всеми персонажами сцены

NPC, определённые как присутствующие в сцене, получают статус `[IN SCENE]` и **полностью исключаются из API-запроса** — на них не тратятся токены, их `lastLocation` не обновляется.

## Система событий

### Масштаб (бросок d20)

Для каждого NPC делается независимый бросок:

| Результат | Масштаб | Значение |
|-----------|---------|----------|
| 1–8 | **MINOR** | Мелкий, забываемый момент. Обычная жизнь. Ничего не меняется навсегда. |
| 9–16 | **NOTABLE** | Реальный сдвиг в жизни персонажа на этой неделе. Что-то, что он запомнит. |
| 17–20 | **MAJOR** | Меняющее жизнь событие. Необратимое или очень значительное. Здоровье, отношения, работа, идентичность. |

### Категории

Пять категорий, выбираются случайно для каждого события:

- **Personal** — внутреннее состояние, тело, привычки, личная жизнь
- **Relationship** — реальный контакт, конфликт, откровение или изменение связи с конкретным человеком
- **Status** — изменение материального, социального, правового или физического положения
- **Discovery** — обнаружение чего-то нового: информации, тайны, истины
- **Social** — участие в групповой динамике, общественной жизни, публичных ситуациях

### Архетипы событий

Каждая комбинация масштаб + категория + тон (позитив/негатив) соответствует пулу из 8 нарративных архетипов. Расширение случайно выбирает один и отдаёт его в промпт как директиву `EVENT ARCHETYPE`. AI получает задание **применить этот архетип к конкретному персонажу** — его личности, отношениям и текущей ситуации — а не генерировать абстрактное событие.

Например, архетип **MAJOR / Relationship / негатив** звучит как:
> "was betrayed by someone at the exact depth where it does the most damage"

AI переводит это в конкретное предложение, подходящее именно этому NPC.

### Формат ответа

AI должен вернуть по одной строке на NPC в точно таком формате:

```
NPC1: локация | minor/notable/major | Описание события.
NPC2: локация | in-scene | No offscreen events. Currently in scene.
```

Расширение парсит это через `parseBatchResponse`. Если AI использует имена персонажей вместо меток `NPC1`, `NPC2` и т.д., срабатывает запасной парсер — он пробует найти совпадение по первому слову имени.

### Самодекларируемый масштаб

AI самостоятельно указывает масштаб в середине строки. Расширение использует именно **этот масштаб** (а не результат броска d20) при решении об автопродвижении в закреплённые факты. Это исключает ситуацию когда мелкое событие оказывается в постоянных фактах только потому что кубик выпал на 17–20.

### Автопродвижение в закреплённые факты

Если AI самостоятельно указывает `major`, это событие автоматически добавляется в список `permanentFacts` NPC. Подробнее — в разделе [Закреплённые факты](#закреплённые-факты).

## Закреплённые факты

Закреплённые факты — это вещи об NPC, которые **всегда истинны** и никогда не должны противоречить сгенерированным событиям. Они хранятся отдельно от скользящей истории событий и инжектируются в каждый промпт как `PERMANENT FACTS (always true, never contradict these)`.

### Как факты добавляются

- **Автоматически** — любое событие, которое AI сам пометил как `major`, автоматически становится фактом
- **Вручную** — кнопка 📌 рядом с любым событием в интерфейсе закрепляет его как постоянный факт

### Как факты удаляются

Кнопка ✕ рядом с фактом в интерфейсе удаляет его.

### Область видимости

Закреплённые факты хранятся отдельно для каждого бота и каждого чата. При создании нового чата с тем же персонажем факты очищаются вместе с событиями (см. [Структура хранилища](#структура-хранилища)).

### Дата истории на фактах

При создании факта (авто или вручную) расширение сохраняет внутриигровые дату и время из текущего инфоблока (например `2008/11/12 14:15`). Это сохраняется вместе с фактом и отображается в интерфейсе для ориентировки на таймлайне.

## Управление NPC

### Сканирование лорбука

Кнопка **Scan Lorebook** читает все лорбуки, прикреплённые к текущему персонажу (через лорбук персонажа, лорбук чата или `chat_metadata`). Находит записи, соответствующие текущему режиму сканирования, и регистрирует их как NPC.

Если NPC уже существует — его описание обновляется, события и факты сохраняются.

### Ручное добавление NPC

NPC можно добавить без записи в лорбуке через раздел **NPC Management → Add new NPC**. Укажите имя и опциональное описание. Добавленные вручную NPC получают флаг `pendingIntro: true` (см. [Ввод нового NPC в сцену](#ввод-нового-npc-в-сцену)).

### Редактирование NPC

Раздел **NPC Management → Edit existing NPC** предоставляет:

- **Notes** (Заметки) — свободный текст, который всегда отправляется в промпт с пометкой `CURRENT NOTES (override lorebook if contradicts)`. Используйте для актуального состояния персонажа: `сейчас беременна`, `в конфликте с Парфёновым`. Заметки хранятся для всех чатов с этим ботом.
- **Description** (Описание) — редактируемая версия описания из лорбука. Оригинальный текст лорбука сохраняется отдельно; кнопка **Reset desc** восстанавливает его.

### Отключение NPC

Кнопка паузы на карточке NPC отключает генерацию событий для этого персонажа без его удаления. Отключённые NPC исключаются из API-запросов.

### Очистка и удаление

- **Корзина на карточке** — удаляет все события и закреплённые факты этого NPC в текущем чате
- **✕ на карточке** — полностью удаляет NPC из всех чатов
- **Clear All Events** — очищает историю событий всех NPC в текущем чате (факты сохраняются)
- **Remove All** — удаляет всех NPC полностью

### Редактирование текста события

Текст события в карточке NPC можно редактировать прямо на месте. Кликните по тексту — появится курсор. После редактирования нажмите Enter или кликните в сторону — изменения сохраняются мгновенно без перерендера карточки. Escape отменяет изменения и восстанавливает оригинал.

## Ввод нового NPC в сцену

При добавлении NPC вручную он получает флаг `pendingIntro: true`. Это влияет на инжекцию двумя способами:

### Пассивный режим

В контекст добавляется тихая строка:
```
[PENDING: Имя — exists in this world, introduce naturally when the moment fits.]
```
Основная модель знает что персонаж существует, но не обязана вводить его прямо сейчас.

### Активный режим (кнопка введения)

Кнопка с иконкой двери на карточке NPC ставит в очередь активное введение. При следующей генерации в контекст добавляется жёсткая директива:

```
[!!! MANDATORY — DO NOT SKIP !!!]
You MUST introduce the following character(s) into THIS response, right now: Имя.
This is not a suggestion. This character MUST physically appear or be directly referenced in the scene...
Do NOT delay it to the next message. Do NOT hint at their arrival. Make it happen NOW.
[!!! END MANDATORY !!!]
```

Это срабатывает один раз и автоматически сбрасывается после следующего ответа бота, независимо от того появился ли персонаж в тексте.

### Автоопределение

После каждого ответа бота расширение проверяет присутствие имени NPC в списке персонажей инфоблока. Если совпадение найдено — флаг `pendingIntro` снимается автоматически, показывается уведомление `"Имя has entered the scene!"`.

## Структура хранилища

Данные хранятся в `extension_settings` SillyTavern под ключом `wild-offscreen`:

```
npcData
└── [botKey]                       ← по аватару или имени персонажа-бота
    ├── __npcs                     ← данные идентичности (общие для всех чатов)
    │   └── [имя]
    │       ├── name
    │       ├── description        ← текущее описание (может быть отредактировано)
    │       ├── lorebookDescription ← оригинал из лорбука (для Reset)
    │       ├── notes              ← поле заметок
    │       ├── enabled
    │       ├── searchKeys
    │       ├── lorebookName
    │       ├── entryUid
    │       └── pendingIntro
    └── [chatKey]                  ← данные времени выполнения (отдельно на каждый чат)
        └── [имя]
            ├── events[]           ← скользящая история событий (макс. N штук)
            │   └── { text, location, scale, category, positive, timestamp, storyDate }
            ├── permanentFacts[]
            │   └── { text, category, positive, timestamp, storyDate, auto }
            └── lastLocation
```

### Новый чат vs возврат к старому

При смене чата расширение сравнивает ключ бота и ключ чата. Если ключ чата новый и данных по нему нет — все события и факты очищаются, показывается уведомление. **Данные идентичности NPC (имена, описания) сохраняются.**

При возврате к старому чату с тем же персонажем все события и факты этого чата восстанавливаются автоматически.

## Справочник функций

### Хранилище и данные

**`getBotKey()`** — Возвращает стабильный идентификатор текущего персонажа-бота на основе имени файла его аватара. Используется как ключ верхнего уровня в хранилище.

**`getChatKey()`** — Возвращает идентификатор текущей сессии чата, позволяя хранить данные о событиях отдельно для каждого чата одного и того же персонажа.

**`getNPCs()`** — Возвращает объединённое представление всех зарегистрированных NPC: данные идентичности из `__npcs` объединяются с данными времени выполнения (события, факты, локация) из текущего чат-хранилища.

**`saveNPCs(npcs)`** — Сохраняет изменённый объект NPC обратно в хранилище, разделяя данные идентичности и данные времени выполнения.

**`deleteNPC(name)`** — Удаляет NPC из `__npcs` и из всех чат-хранилищ текущего бота.

**`clearChatData()`** — Удаляет все данные времени выполнения (события, факты, локации) для всех NPC в текущем чате. Данные идентичности не затрагиваются.

### Сканирование лорбука

**`isNPCEntry(entry, mode)`** — Возвращает true если запись лорбука подходит как источник NPC в указанном режиме сканирования.

**`extractNPCInfo(entry, bookName)`** — Извлекает имя, описание, поисковые ключи и метаданные из объекта записи лорбука.

**`scanCharacterLorebooks()`** — Находит все лорбуки прикреплённые к текущему персонажу и сканирует их на предмет записей NPC.

**`registerNPCs(scanned)`** — Принимает список найденных NPC и добавляет новых в хранилище (или обновляет описания существующих).

### Определение сцены

**`normalizeDateStr(str)`** — Преобразует даты с месяцем-словом (`2008/Ноябрь/12`) в числовой формат (`2008/11/12`). Поддерживает русский (все падежи) и английский.

**`hasDatePattern(str)`** — Возвращает true если строка содержит дату в числовом или словесном формате.

**`parseSceneInfo()`** — Находит последнее сообщение с инфоблоком и возвращает `{ date, time, season, location, characters[] }`. Это основной источник информации о том кто присутствует в активной сцене.

**`parseInfoblockChars(mes)`** — Парсит список персонажей из инфоблока одного сообщения, возвращает массив строк с именами.

**`parseKeyAsRegex(key)`** — Если строка ключа имеет формат `/паттерн/флаги` — компилирует и возвращает RegExp. Иначе возвращает null.

**`npcInInfoblock(npc, mes)`** — Возвращает true если NPC присутствует в списке персонажей инфоблока сообщения: проверяет прямое совпадение имени, обычные ключи и ключи-регексы.

### Построение контекста

**`getMainCharInfo()`** — Возвращает текстовый блок с описанием персонажа-бота и персоны пользователя, из карточки персонажа и настроек активной персоны.

**`getChatContextForNPC(npc, maxMessages, maxCharsPerMsg)`** — Возвращает недавние сообщения чата, где этот NPC присутствовал в инфоблоке. Используется для предоставления AI релевантного контекста сцены по конкретному персонажу. Если сцен с этим NPC нет — возвращает последние N сообщений.

### Генерация событий

**`rollEventParams()`** — Бросает d20, определяет масштаб (minor/notable/major), выбирает случайную категорию и тональность (позитив/негатив), возвращает объект params.

**`pickArchetype(scaleId, category, isPositive)`** — Возвращает случайный архетип из `EVENT_POOLS[scaleId][category][positive/negative]`.

**`buildBatchMessages(npcList, mainCharInfo, sharedChatContext, sceneInfo)`** — Строит полный массив сообщений для пакетного запроса к API. Помечает каждого NPC как `[IN SCENE]` или `[OFFSCREEN]`, включает заголовок сцены с датой/временем/локацией, закреплённые факты, историю событий, заметки и директиву архетипа.

**`callAPI(messages, npcCount)`** — Отправляет массив сообщений на эндпоинт выбранного Connection Profile. `max_tokens` рассчитывается динамически: `max(400, npcCount × 120)`.

**`parseBatchResponse(text, count, npcList)`** — Парсит ответ AI. Ожидает строки в формате `NPC1: локация | масштаб | предложение`. При отсутствии меток NPC пробует найти совпадение по имени персонажа. Возвращает массив объектов `{ location, text, inScene, reportedScale }`.

**`generateEventsForAllNPCs(npcs)`** — Оркестрирует полный цикл генерации: фильтрует включённых NPC, исключает присутствующих в сцене, строит промпт, вызывает API, парсит результаты, добавляет события, обрабатывает автопродвижение в постоянные факты, сохраняет всё.

**`runGenerationCycle()`** — Вызывается по триггеру сообщения. Защищён от повторного входа через флаг `isGenerating`. Запускает `generateEventsForAllNPCs`, перечитывает данные после сохранения для точного подсчёта результатов, показывает уведомления.

### Инжекция контекста

**`buildInjectionText(npcs, injectMax)`** — Строит текстовый блок `[OFF-SCREEN NPC UPDATES]`, инжектируемый в основной контекст чата. Включает последнюю известную локацию и последние события каждого включённого NPC, а также инструкции по введению новых персонажей.

**`updateInjection()`** — Вызывает `buildInjectionText` и передаёт результат в `setExtensionPrompt`. Также обновляет счётчик токенов в интерфейсе (`~N tokens injected`).

### Интерфейс

**`renderNPCList()`** — Перестраивает список карточек NPC. Запоминает какие карточки были открыты перед перерендером и восстанавливает их состояние — например, закрепление факта не схлопывает карточку.

**`populateEditSelect()`** — Заполняет выпадающий список выбора NPC в разделе NPC Management.

**`buildUI()`** — Создаёт полный HTML панели расширения с четырьмя раскрывающимися секциями.

**`onBotMessageDone()`** — Общий обработчик для событий `CHARACTER_MESSAGE_RENDERED` и `MESSAGE_RECEIVED` (оба отслеживаются для надёжности). Дедуплицируется по ID сообщения. Обрабатывает определение реролла, автоматически снимает `pendingIntro` когда NPC появляется в инфоблоке, увеличивает счётчик `msgCounter`, запускает генерацию когда счётчик достигает `triggerEvery`.

## Справочник настроек

| Настройка | По умолчанию | Описание |
|-----------|-------------|----------|
| `enabled` | `true` | Главный переключатель вкл/выкл |
| `triggerEvery` | `5` | Генерировать события каждые N сообщений (учитываются и сообщения бота, и пользователя) |
| `maxEvents` | `7` | Максимальное количество хранимых событий на NPC в одном чате |
| `maxMessages` | `30` | Сколько последних сообщений читать для контекста чата |
| `maxCharsPerMsg` | `2000` | Ограничение символов на сообщение при построении контекста |
| `maxTokens` | `200` | Базовое значение max_tokens (реальное: `max(400, количество_NPC × 120)`) |
| `connectionProfile` | `''` | Имя Connection Profile для API-вызовов |
| `outputLanguage` | `'en'` | Язык генерируемых событий: `en`, `ru` или `uk` |
| `scanPosition` | `'before_char'` | Позиция записей лорбука для сканирования: `before_char` или `after_char` |

## Разделы интерфейса

Панель расширения состоит из четырёх раскрывающихся секций:

### Characters (открыта по умолчанию)
- Текущие дата и время из последнего инфоблока
- Список карточек NPC с событиями и закреплёнными фактами
- **Generate Now** — немедленный запуск цикла генерации
- **Scan Lorebook** — сканирование прикреплённых лорбуков
- **Clear All Events** — очистка истории событий всех NPC в текущем чате (факты сохраняются)
- **Remove All** — полное удаление всех NPC

### Lorebook & Context
- Имя лорбука и выбор позиции сканирования
- Глубина истории сообщений
- Максимум символов на сообщение
- Частота генерации событий и максимум хранимых событий

### NPC Management
- Добавление нового NPC вручную (имя + описание)
- Редактирование существующего NPC: выбор из списка, редактирование заметок и описания, сброс описания к оригиналу из лорбука

### API & Language
- Выбор языка вывода (EN / RU / UK)
- Выбор Connection Profile с кнопкой обновления

## Нюансы и поведение

**Определение реролла** — При реролле сообщения расширение замечает что длина чата не изменилась и удаляет последнее событие у каждого NPC перед повторной генерацией. Закреплённые факты при реролле **не затрагиваются никогда**.

**Новый чат vs возврат к старому** — Расширение отслеживает ID чатов. Возврат к предыдущему чату с тем же персонажем восстанавливает все события и факты из той сессии.

**NPC в сцене полностью исключаются** — Они фильтруются до отправки API-запроса. Никаких токенов на них не тратится, их `lastLocation` не меняется.

**Запасной парсер** — Если AI отвечает `Илья: школьный коридор | minor | ...` вместо `NPC1: ...`, парсер пробует найти совпадение по первому слову имени. В консоли появится запись `Fallback name match for NPC1 (Илья)`.

**Двойной триггер генерации** — Отслеживаются оба события `CHARACTER_MESSAGE_RENDERED` и `MESSAGE_RECEIVED`. Дедупликация по ID сообщения гарантирует ровно один запуск на сообщение.

**Язык вывода** — Настройка языка добавляет одну строку в системный промпт (`Write in Russian.` и т.д.) и влияет только на текст событий, но не на служебные метки вроде `NPC1:` или слов масштаба.

**Счётчик токенов** — Отображение `~N tokens injected` в интерфейсе — приблизительная оценка на основе длины текста, делённой на 4 (подходит для смешанного русского/английского текста).

**Заметки vs Описание** — Описание — полный бэкграунд персонажа для AI. Заметки — краткое актуальное состояние с более высоким приоритетом. Используйте заметки для вещей, меняющихся по ходу истории: `сейчас под следствием`, `скрывает что-то от Парфёнова`.

**Поисковые ключи и определение сцены** — Если имя персонажа в лорбуке написано по-английски, а в инфоблоке по-русски — добавьте русские формы имени (обычные или regex) в массив `key` записи лорбука. Расширение будет использовать их для определения присутствия в сцене.
